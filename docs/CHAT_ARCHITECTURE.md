# Архитектура чата dedART

## Обзор

Чат в dedART построен по принципу MVC (Model-View-Controller) с соблюдением принципов SOLID и лучших практик ООП.

## Структура

```
src/chat/
├── models/           # Модели данных
│   ├── chat_message.py      # Модель сообщения
│   ├── conversation.py      # Модель разговора
│   └── ai_response.py       # Модель ответа ИИ
├── views/            # Представления (UI)
│   ├── chat_view.py         # Основное представление чата
│   ├── message_widget.py    # Виджет отдельного сообщения
│   └── input_widget.py      # Виджет ввода сообщений
├── controllers/      # Контроллеры
│   └── chat_controller.py   # Основной контроллер чата
├── services/         # Сервисы
│   ├── ai_service.py        # Сервис работы с ИИ
│   ├── cache_service.py     # Сервис кеширования
│   └── command_processor.py # Обработчик команд
└── utils/            # Утилиты
    ├── message_formatter.py # Форматирование сообщений
    └── chat_logger.py       # Логирование чата
```

## Принципы архитектуры

### SOLID принципы

1. **Single Responsibility Principle (SRP)**
   - Каждый класс отвечает за одну задачу
   - `ChatMessage` - только за хранение данных сообщения
   - `ChatController` - только за координацию между компонентами
   - `AIService` - только за взаимодействие с ИИ

2. **Open/Closed Principle (OCP)**
   - Классы открыты для расширения, закрыты для модификации
   - Новые типы сообщений можно добавить через наследование
   - Новые ИИ сервисы реализуют базовый интерфейс

3. **Liskov Substitution Principle (LSP)**
   - Наследники полностью заменяют базовые классы
   - `MockAIService` и `OpenAIService` взаимозаменяемы

4. **Interface Segregation Principle (ISP)**
   - Интерфейсы содержат только необходимые методы
   - `BaseAIService` предоставляет минимальный интерфейс

5. **Dependency Inversion Principle (DIP)**
   - Зависимости от абстракций, а не от конкретных реализаций
   - `ChatController` зависит от `BaseAIService`, а не от конкретной реализации

### MVC паттерн

- **Model**: Хранение и логика данных (`ChatMessage`, `Conversation`, `AIResponse`)
- **View**: Отображение данных (`ChatView`, `MessageWidget`, `InputWidget`)  
- **Controller**: Координация между Model и View (`ChatController`)

## Основные компоненты

### Модели (Models)

#### ChatMessage
- Хранит данные одного сообщения
- Поддерживает различные типы (текст, код, изображение)
- Имеет роли (пользователь, ассистент, система)

#### Conversation
- Управляет коллекцией сообщений
- Предоставляет методы поиска и фильтрации
- Поддерживает сериализацию/десериализацию

#### AIResponse
- Содержит ответ от ИИ
- Хранит метаданные (время обработки, информация о модели)
- Может содержать команды для выполнения

### Представления (Views)

#### ChatView
- Основной UI компонент чата
- Отображает список сообщений
- Содержит область ввода

#### MessageWidget
- Отображает отдельное сообщение
- Поддерживает различные стили для разных ролей
- Предоставляет действия (копирование, редактирование)

#### InputWidget
- Область ввода сообщений
- Поддерживает многострочный ввод
- Содержит кнопки управления

### Контроллер (Controller)

#### ChatController
- Координирует взаимодействие между компонентами
- Обрабатывает отправку сообщений
- Управляет состоянием чата
- Обрабатывает команды от ИИ

### Сервисы (Services)

#### AIService
- Абстрактный интерфейс для работы с ИИ
- `MockAIService` - демо-реализация
- `OpenAIService` - заготовка для OpenAI API

#### CacheService
- Кеширование разговоров в памяти и на диске
- Управление метаданными
- Поиск по разговорам

#### CommandProcessor
- Обработка команд от ИИ
- Поддержка различных типов команд
- Расширяемая архитектура обработчиков

### Утилиты (Utils)

#### MessageFormatter
- Форматирование сообщений для отображения
- Поддержка Markdown
- Экспорт в различные форматы

#### ChatLogger
- Специализированное логирование чата
- Статистика сессий
- Детальные логи в JSON формате

## Потоки данных

### Отправка сообщения

1. Пользователь вводит текст в `InputWidget`
2. `ChatView` получает сигнал и передает в `ChatController`
3. `ChatController` создает `ChatMessage` и добавляет в `Conversation`
4. Сообщение отображается через `MessageWidget`
5. `ChatController` отправляет запрос в `AIService`
6. Получен `AIResponse`, создается сообщение ассистента
7. Ответ отображается в UI

### Обработка команд

1. `AIResponse` содержит команды
2. `CommandProcessor` обрабатывает команды
3. Соответствующие обработчики выполняют действия
4. Результаты логируются

## Расширение функциональности

### Добавление нового типа сообщений

1. Добавить значение в `MessageType` enum
2. Обновить `MessageFormatter` для нового типа
3. При необходимости создать специальный виджет

### Добавление нового ИИ провайдера

1. Наследовать от `BaseAIService`
2. Реализовать методы `generate_response`, `is_available`, `get_model_info`
3. Добавить в `AIServiceFactory`

### Добавление нового типа команд

1. Добавить значение в `CommandType` enum
2. Создать обработчик, наследующий от `BaseCommandHandler`
3. Зарегистрировать в `CommandProcessor`

## Тестирование

Архитектура позволяет легко тестировать компоненты:

- Модели можно тестировать изолированно
- Контроллер можно тестировать с mock-сервисами
- UI компоненты можно тестировать с mock-контроллером

## Производительность

- Асинхронная обработка запросов к ИИ
- Кеширование разговоров
- Ленивая загрузка сообщений
- Эффективное управление памятью

- Валидация входных данных
- Экранирование HTML в сообщениях
- Логирование всех действий
